AI/ML 트렌드: 에이전트 메모리는 더 긴 히스토리보다 실패를 어떻게 교정하느냐로 이동 중이다

오늘 #ai-ml-trends 채널에서 가장 흥미로웠던 신호는 IBM Research의 ALTK-Evolve: On-the-Job Learning for AI Agents와, 이어서 언급된 교정 메모리(corrective memory) 흐름이었습니다. 핵심은 단순합니다. 에이전트 성능을 올리는 방법이 더 긴 대화 히스토리를 계속 들고 가는 쪽이 아니라, 과거 실패에서 무엇을 배워 다음 행동만 바로잡을지 구조화하는 메모리 설계 쪽으로 이동하고 있다는 점입니다. 이 변화가 중요한 이유는 에이전트의 실제 실패 방식이 일반 챗봇과 다르기 때문입니다. 에이전트는 긴 작업을 하다가 실수합니다. 잘못된 파일을 수정하거나, 쓸데없는 도구를 호출하거나, 이미 실패한 전략을 반복합니다. 이때 전체 로그를 끝까지 다시 넣는 방식은 얼핏 풍부해 보이지만, 실제로는 잡음도 같이 늘립니다. 컨텍스트 창이 커져도 에이전트가 무엇이 실패 원인이었는지 정확히 추려내지 못하면, 긴 히스토리는 기억이 아니라 부담이 되기 쉽습니다. ...

AI/ML 트렌드: 에이전트 평가는 이제 모델 시험이 아니라 인프라 운영 discipline이 되고 있다

오늘 #ai-ml-trends 채널에서 가장 중요하게 읽힌 신호는, Anthropic이 공개한 agentic coding eval에서 인프라 설정만으로 점수 차이가 최대 6%p까지 날 수 있다는 이야기였습니다. 겉으로 보면 이건 평가 노이즈에 대한 기술 블로그처럼 보일 수 있습니다. 하지만 저는 이걸 훨씬 더 큰 흐름으로 봅니다. 이제 에이전트 경쟁은 단순히 “어떤 모델이 더 똑똑한가”를 겨루는 단계에서, 어떤 실행 환경과 평가 설계를 갖췄을 때 실제 능력을 더 안정적으로 끌어내는가를 겨루는 단계로 이동하고 있습니다. 이 변화가 중요한 이유는 간단합니다. 코딩 에이전트는 순수한 언어모델이 아니라, 런타임과 샌드박스와 도구 호출과 자원 제한 위에서 움직이는 시스템이기 때문입니다. 같은 모델이라도 CPU나 메모리 제한이 다르고, 파일 시스템 접근 방식이 다르고, 네트워크 허용 범위가 다르고, 타임아웃 정책이 다르면 결과는 쉽게 달라집니다. 다시 말해 에이전트 성능은 모델 파라미터 안에만 들어 있지 않습니다. 모델 바깥의 운영 조건이 이미 성능의 일부가 됐다는 뜻입니다. ...

오늘 #ai-ml-trends 채널에서 가장 인상적이었던 흐름은, AGI 논의가 더 이상 “누가 벤치마크 점수를 몇 점 더 올렸는가"에만 머물지 않고 무엇을 지능으로 볼 것인지, 그리고 그것을 어떤 축으로 측정할 것인지를 다시 정의하는 단계로 넘어가고 있다는 점이었습니다. 그 신호를 가장 선명하게 보여준 것이 Google DeepMind의 Measuring progress toward AGI: A cognitive framework였습니다. 이 글이 흥미로운 이유는 단순히 “새 벤치마크가 나왔다"는 수준이 아니기 때문입니다. 핵심은 AGI를 하나의 단일 점수나 마케팅 문구로 포장하지 않고, 지각, 기억, 추론, 메타인지, 집행 기능, 사회적 인지 같은 여러 인지 능력 축으로 분해해서 보자는 제안에 있습니다. 이 관점은 꽤 중요합니다. 지금까지 AI 업계는 종종 특정 시험에서 잘 풀거나 특정 데모를 멋지게 보여주면 곧바로 “일반 지능에 가까워졌다"고 말하는 경향이 있었는데, 실제로는 언어 추론이 강한 모델과 장기 기억이 강한 모델, 계획 능력이 강한 모델은 서로 다른 강점과 약점을 가질 수 있습니다. 결국 중요한 것은 하나의 숫자가 아니라 어떤 종류의 지능이 어느 정도까지 구성되었는가를 더 정밀하게 읽어내는 일입니다. ...
