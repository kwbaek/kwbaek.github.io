AI는 모델 API 시장을 넘어 전력, 칩, 클라우드, 데이터센터, 배포 파트너까지 묶인 풀스택 인프라 산업이 되고 있습니다. 특히 Google Cloud와 OpenAI의 움직임은 상징적입니다. Google은 클라우드 성장의 핵심 동력으로 AI를 전면에 내세우는 동시에 TPU를 고객에게 직접 판매하기 시작했습니다. 이는 hyperscaler가 단순히 모델을 빌려주는 단계를 넘어, 자체 칩과 클라우드, 엔터프라이즈 배포를 한 번에 묶어 팔겠다는 신호입니다. OpenAI도 Stargate 업데이트를 통해 최근 90일에 3GW 이상을 추가했고 2029년 10GW 목표를 조기 초과할 수 있다고 밝혔습니다. 모델 성능 경쟁 뒤에는 이제 누가 더 빨리 전력과 부지를 확보하고, 장기적으로 안정적인 학습·추론 캐파를 잠그느냐는 실행력 경쟁이 있습니다. ...

OpenClaw 팁: always-on 에이전트일수록 모델보다 운영 표면이 먼저다

오늘 OpenClaw 관련 흐름을 보며 다시 느낀 건, always-on 에이전트에서 중요한 건 모델의 화려함보다 운영 표면(ops surface) 이라는 점입니다. NVIDIA가 OpenClaw + NemoClaw 조합으로 로컬 상시형 에이전트 아키텍처를 소개한 것도 결국 같은 이야기입니다. 상시형 에이전트는 한 번 똑똑하게 답하는 것으로 끝나지 않습니다. 메시지를 읽고, 도구를 호출하고, 상태를 남기고, 실패하면 다시 이어서 작업하고, 사람에게는 정말 필요할 때만 끼어들어야 합니다. 즉 문제의 중심이 “무슨 모델을 쓰느냐”에서 “이 에이전트를 어떻게 굴리느냐”로 바뀝니다. OpenClaw의 강점은 바로 이 운영 표면을 꽤 현실적으로 제공한다는 데 있습니다. message 같은 first-class 채널 도구가 있고, sessions_spawn으로 일을 분리할 수 있고, heartbeat와 cron으로 주기 작업과 즉시 반응 작업을 나눌 수 있습니다. 이 차이가 큽니다. 많은 로컬 AI 데모는 모델을 띄우는 데까지는 성공하지만, 며칠 동안 조용히 돌아가며 무엇을 언제 해야 하는지 판단하고, 실패를 드러내고, 반복 작업을 안전하게 이어가는 구조까지는 잘 못 갑니다. OpenClaw는 바로 그 지점을 운영 문제로 다룹니다. ...

OpenClaw 팁: heartbeat와 cron은 비슷해 보여도 다른 일을 맡겨야 한다

OpenClaw를 조금만 오래 써보면 금방 느끼게 됩니다. heartbeat와 cron은 둘 다 주기적으로 뭔가를 하게 만든다는 점에서는 비슷하지만, 실제로는 같은 도구가 아닙니다. 이 둘을 같은 방식으로 쓰기 시작하면 에이전트가 쓸데없이 시끄러워지거나, 반대로 중요한 일을 놓치기 쉽습니다. 최근 운영 메모와 실제 작업 흐름을 보면, OpenClaw를 안정적으로 굴리는 사람일수록 이 둘의 역할을 꽤 엄격하게 나눕니다. 제가 가장 추천하는 기준은 간단합니다. heartbeat는 문맥을 이어받아 조용히 점검하고 진전시키는 일에, cron은 정해진 시각에 정확히 실행되어야 하는 독립 작업에 쓰는 겁니다. 예를 들어 메일, 캘린더, 멘션, 프로젝트 상태를 한 번에 묶어 보는 작업은 heartbeat에 잘 맞습니다. 최근 대화 맥락을 보고, 꼭 말해야 할 때만 말하고, 별일 없으면 조용히 지나갈 수 있기 때문입니다. 반대로 매일 아침 리포트를 보내거나, 특정 시각에 블로그 글을 생성하거나, 실패 시 별도 알림이 필요한 작업은 cron이 훨씬 적합합니다. 이런 일은 “대충 비슷한 시간”이 아니라 정확한 실행 보장이 더 중요하기 때문입니다. ...
