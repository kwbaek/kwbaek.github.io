Hermes/리오 팁: 에이전트 코드리뷰를 감사 가능한 작업으로 만들기

Hermes/리오로 코드 작성, 리뷰, 블로그 배포 같은 반복 작업을 자동화할 때 가장 위험한 착각은 “에이전트가 맞는 말을 했으니 충분하다”입니다. 실제 운영에서는 답변의 그럴듯함보다 무엇을 읽었고, 어떤 변경을 했고, 어떤 검증을 통과했는지가 더 중요합니다. 특히 코드리뷰 자동화는 PR에 코멘트를 다는 순간 팀의 merge 판단에 영향을 줍니다. 그래서 Hermes/리오 워크플로우에서는 코드리뷰를 단순한 자연어 의견이 아니라 감사 가능한 작업 단위로 설계하는 편이 좋습니다. 첫 번째 원칙은 입력 범위를 명확히 남기는 것입니다. 에이전트가 리뷰한 diff, 관련 파일, 테스트 결과, 최근 커밋, 사용자 지시를 구분해야 합니다. “전체 repo를 봤다”는 표현은 거의 항상 부정확합니다. 실제로는 특정 파일 몇 개, 특정 로그, 특정 테스트 출력만 봤을 가능성이 큽니다. 리뷰 결과에는 어떤 근거를 사용했는지 짧게라도 남겨야 합니다. 이렇게 하면 나중에 잘못된 판단이 나왔을 때 모델 탓만 하지 않고, 입력이 부족했는지, 검증이 빠졌는지, 규칙이 애매했는지 확인할 수 있습니다. ...
