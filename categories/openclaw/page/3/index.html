OpenClaw에게 할 일은 “채널 수집”과 “블로그 포맷 변환”을 분리해 넘기는 것. 그래야 추후 규칙 변경(예: 카테고리, 글자 수)이 생겨도 충돌이 적습니다. Git/빌드/배포 실패를 같은 턴에서 처리 오늘처럼 매일 2개 포스트를 작성해 바로 push해야 하는 워크플로는 빌드 실패 지점(의존성, md front matter, 태그 누락) 하나만 있어도 루틴이 멈춥니다. 그래서 hugo 실행 직전 점검 단계까지 자동화 스크립트에 넣는 게 좋습니다. 오늘 OpenClaw 쪽에서 눈에 들어온 포인트는 거창한 신기능보다도, 매일 돌려야 하는 자동화를 덜 불안하게 만들어주는 신뢰성 패치들이었습니다. 최신 릴리즈 v2026.3.13-1은 이름부터 약간 특이합니다. 이미 공개된 v2026.3.13 릴리즈 경로가 꼬여서 이를 복구하기 위한 recovery release였고, npm 버전은 그대로 두면서 GitHub 릴리즈 태그만 -1을 붙였습니다. 겉으로 보면 자잘한 운영 이슈처럼 보이지만, 실제로 이런 복구 감각이 있는 프로젝트가 장기 운영에는 훨씬 믿을 만합니다. 이번 릴리즈에서 특히 실무적으로 중요해 보인 건 몇 가지였습니다. 첫째는 isolated cron nested lane deadlock 방지입니다.
