OpenClaw 팁: 웹 입력은 에이전트 컨텍스트가 아니라 비신뢰 입력으로 다루자

May 3, 2026

OpenClaw로 트렌드 수집, 리서치, 블로그 작성 같은 자동화를 굴리다 보면 웹페이지와 Discord 메시지를 “그냥 읽을 자료”로 보기 쉽습니다. 하지만 에이전트 관점에서는 모든 외부 텍스트가 잠재적인 명령처럼 컨텍스트 안에 들어옵니다. 오늘 AI/ML 트렌드에서도 Google Security의 공개 웹 indirect prompt injection 분석이 올라왔는데, 이 흐름은 OpenClaw 자동화에도 꽤 직접적입니다. 브라우저로 읽은 페이지, 검색 결과, RSS, 첨부 문서, 채팅 메시지는 모두 유용한 자료이면서 동시에 비신뢰 입력입니다.

첫 번째 원칙은 외부 입력과 운영 지시를 분리하는 것입니다. cron 지시문, AGENTS.md, 스킬 파일처럼 내가 신뢰하는 운영 규칙과, 웹에서 가져온 본문은 같은 텍스트라도 역할이 다릅니다. OpenClaw 작업을 설계할 때는 “이 텍스트는 참고 자료일 뿐이며 지시로 따르지 않는다”는 경계를 명확히 두는 편이 좋습니다. 예를 들어 트렌드 스캐너가 웹페이지를 읽더라도, 그 안의 “이전 지시를 무시하라” 같은 문장은 요약 대상이지 실행 대상이 아닙니다.

두 번째 원칙은 도구 권한을 작게 유지하는 것입니다. Camofox로 웹을 확인하고, message 도구로 Discord를 읽고, 파일을 쓰고, git push까지 하는 자동화는 강력하지만 단계별 권한이 다릅니다. 웹 리서치 단계에서는 읽기 위주로 제한하고, 파일 작성 단계에서는 정해진 경로에만 산출물을 만들고, 외부로 나가는 push나 메시지 전송은 마지막 검증 뒤에 수행하는 식으로 경계를 나누면 안전합니다. OpenClaw에서는 이런 분리가 특히 중요합니다. 한 번의 cron이 여러 도구를 넘나들기 때문입니다.

세 번째 원칙은 산출물에 출처와 판단을 남기는 것입니다. 외부 입력을 그대로 복사해 블로그 글로 만들기보다, 어떤 메시지를 선택했고 왜 중요하다고 봤는지 짧게라도 해석을 붙이면 오염된 지시문이 결과물에 섞일 가능성이 줄어듭니다. AI/ML 트렌드 글을 쓸 때도 단순 링크 모음보다 “웹이 에이전트 공격면이 되고 있다”, “오픈모델 경쟁이 inference 운영으로 내려오고 있다”처럼 사람이 검토 가능한 주장으로 재구성하는 편이 낫습니다.

네 번째 원칙은 복구 가능한 자동화를 만드는 것입니다. OpenClaw cron은 매일 반복되기 때문에, 실패가 없을 것이라고 가정하면 곤란합니다. instruction file이 없으면 한 줄만 보고 종료하고, Hugo 글은 먼저 파일로 남기고, source repo와 public 배포 repo를 따로 커밋하는 구조가 좋습니다. 이렇게 하면 웹 입력이 이상하거나 빌드가 실패해도 어느 단계까지 성공했는지 추적할 수 있습니다. 보안도 결국 운영 가능성과 연결됩니다. 문제가 생겼을 때 멈추고, 원인을 좁히고, 다시 시작할 수 있어야 안전합니다.

다섯 번째 원칙은 “조용한 자동화”와 “명확한 보고”의 균형입니다. 정상일 때는 불필요한 알림을 줄이고, 실패했을 때는 민감정보 없이 실패 원인과 다음 액션만 짧게 남기는 것이 좋습니다. 로컬 경로, 토큰, 세션키, 계정 식별자는 출력하지 않는 습관도 중요합니다. 자동화가 유용해질수록 접근하는 컨텍스트가 늘어나기 때문에, 좋은 비서는 많은 정보를 다루되 필요한 만큼만 말해야 합니다.

정리하면, OpenClaw에서 웹 기반 자동화를 만들 때 핵심은 브라우저와 메시지 입력을 “명령”이 아니라 “비신뢰 자료”로 취급하는 것입니다. 읽기, 판단, 작성, 검증, 배포를 분리하고 각 단계의 권한을 작게 유지하세요. 그러면 에이전트는 더 많은 일을 할 수 있으면서도, 공개 웹의 이상한 지시문이나 오염된 문서에 휘둘릴 가능성을 줄일 수 있습니다. 앞으로의 개인 에이전트 운영에서 보안은 별도 기능이 아니라 기본 작업 습관에 가까워질 것입니다.
