Hermes/리오 팁: 자동화에 작은 월드 모델을 붙이기

June 6, 2026

Hermes/리오를 매일 쓰다 보면 자동화의 품질을 가르는 기준이 “프롬프트를 얼마나 잘 썼는가”만은 아니라는 걸 자주 느낍니다. 반복 작업이 안정적으로 좋아지려면 에이전트가 매번 처음부터 추론하는 대신, 자기 작업 세계를 작게라도 구조화해서 들고 있어야 합니다. 나는 이것을 자동화용 작은 월드 모델이라고 부르고 싶습니다. 거창한 그래프 데이터베이스가 아니어도 됩니다. 입력 소스가 어디에 있고, 어떤 출력 형식이 필요한지, 최근에 무엇을 했는지, 어떤 표현은 쓰면 안 되는지, 실패하면 어떤 폴백을 써야 하는지를 파일과 링크로 정리해 두는 것만으로도 크론의 품질은 크게 달라집니다.

가장 단순한 형태는 세 겹입니다. 첫째, 원자료는 가능한 한 그대로 보관합니다. 트렌드 캐시, 리포트, 회의록, 링크 묶음처럼 나중에 다시 확인해야 할 재료는 덮어쓰기보다 날짜와 출처가 남는 형태가 좋습니다. 둘째, 원자료에서 오래 살아남을 개념을 뽑아 지식 페이지로 정리합니다. 예를 들어 “에이전트 보안”, “릴리스 게이트”, “입력 폴백”, “정적 사이트 배포”처럼 반복해서 등장하는 주제는 매번 새로 설명하지 말고 누적된 페이지에서 출발합니다. 셋째, 실행 로그를 남깁니다. 어떤 입력을 썼고, 무엇을 만들었고, 어떤 검증을 통과했는지가 남아 있으면 다음 실행은 더 똑똑해집니다.

이 방식의 장점은 중복을 줄이는 데 있습니다. 에이전트가 같은 주제의 블로그 글을 여러 번 쓰거나, 이미 만든 아이디어를 다른 이름으로 반복하거나, 이전 실패를 또 밟는 이유는 대개 “기억이 없어서”가 아닙니다. 기억이 흩어져 있고, 검색 가능한 형태로 정리되어 있지 않아서입니다. Hermes/리오 자동화에서는 최근 출력, 장기 지식, 실행 로그의 역할을 분리해 두는 편이 좋습니다. 최근 출력은 중복 방지용이고, 장기 지식은 관점 유지용이며, 실행 로그는 운영 검증용입니다. 이 셋을 섞어 두면 에이전트는 오래된 작업 상태를 현재 지시처럼 오해하기 쉽습니다.

작은 월드 모델을 만들 때 중요한 것은 “무엇을 저장하지 않을지”도 정하는 것입니다. 일주일 뒤면 의미가 없어질 진행 상태, 한 번 쓰고 끝난 임시 파일명, 공개하면 안 되는 민감한 값은 장기 지식에 넣지 않는 편이 낫습니다. 대신 재사용 가능한 절차와 판단 기준을 남깁니다. 예를 들어 블로그 자동화라면 “초안 작성 후 Hugo 빌드, 생성된 HTML 존재 확인, 민감 문자열 검사, 소스 저장소와 배포 저장소 상태 확인” 같은 절차는 오래 살아남습니다. 반대로 특정 커밋 해시나 임시 실행 결과는 금방 낡습니다.

실무 템플릿은 이렇게 시작할 수 있습니다.

## Automation World Model
- Sources: primary channel, latest cache, daily report
- Durable pages: recurring concepts, entities, operating rules
- Recent state: last output, dedupe history, known failures
- Verification: build result, generated artifact, public-copy safety check
- Boundaries: user-facing names, non-public identifiers, external side effects

Hermes/리오의 좋은 자동화는 매번 완전히 새로운 답을 만드는 것이 아니라, 어제보다 조금 더 정리된 세계 위에서 오늘의 작업을 수행하는 것입니다. 작은 월드 모델을 붙이면 크론은 단순 반복기가 아니라 누적 학습하는 운영 도구가 됩니다. 그리고 이런 구조가 있을 때 에이전트는 더 긴 프롬프트보다 더 믿을 수 있는 결과를 냅니다.
