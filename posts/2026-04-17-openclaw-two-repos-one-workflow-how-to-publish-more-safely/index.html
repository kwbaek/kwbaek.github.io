OpenClaw 팁: 콘텐츠 레포와 배포 레포를 나누면 자동 발행이 훨씬 덜 무너진다

April 17, 2026

OpenClaw로 반복 발행 작업을 굴릴수록 더 확실해지는 운영 원칙이 하나 있습니다. 블로그 자동화는 “글을 쓰고 끝”이 아니라, 사실상 콘텐츠 생성과 배포 산출물 관리라는 서로 다른 두 가지 책임을 함께 다루는 일입니다. 이 둘을 한 덩어리로 취급하면 처음엔 편해 보여도, 조금만 작업이 쌓이면 금방 어디서 깨졌는지 추적하기 어려워집니다. 반대로 콘텐츠 레포와 public/ 배포 레포를 명확히 나누면, OpenClaw 같은 에이전트형 워크플로에서 실패 지점을 훨씬 더 또렷하게 관리할 수 있습니다.

이 구조가 특히 좋은 이유는 문제를 단계별로 끊어 볼 수 있기 때문입니다. 예를 들어 일일 블로그 포스팅 자동화를 생각해 보면 실제로는 아래 세 단계가 따로 존재합니다.

  1. 오늘의 주제를 읽고 글 원본을 만든다.
  2. Hugo 빌드가 실제로 성공하는지 확인한다.
  3. 생성된 정적 파일을 배포 대상에 반영한다.

겉보기엔 당연한 이야기 같지만, 자동화가 불안정해지는 곳은 늘 이 경계가 흐려질 때입니다. 글 파일은 생겼는데 front matter가 잘못돼 빌드가 깨질 수도 있고, 빌드는 성공했는데 public/ 쪽 반영이 빠질 수도 있습니다. 또는 커밋은 됐지만 어느 레포의 어떤 변경이 실제 문제였는지 나중에 헷갈릴 수도 있습니다. 저는 이런 상황을 줄이는 데 OpenClaw의 단계적 실행 방식이 꽤 잘 맞는다고 느꼈습니다.

핵심은 각 단계를 별도 책임으로 다루는 것입니다.

  • content/posts에 새 글을 쓰는 단계는 편집 책임입니다.
  • hugo를 돌리는 단계는 검증 책임입니다.
  • public/을 커밋하고 푸시하는 단계는 배포 책임입니다.

이렇게 나누면 이점이 분명합니다. 먼저, 실패 원인을 좁히기가 쉬워집니다. 글이 문제인지, 템플릿이 문제인지, 배포 산출물이 문제인지 금방 드러납니다. 둘째, 재실행도 훨씬 편해집니다. 글은 이미 괜찮고 빌드만 다시 하면 되는 상황에서 처음부터 전부 다시 돌릴 필요가 없습니다. 셋째, 에이전트가 작업을 이어받을 때도 맥락이 단순해집니다. “지금 어디까지 끝났는가”가 명확하니까요.

OpenClaw 관점에서 보면 이건 단순한 git 습관이 아니라, 에이전트 친화적인 운영 설계에 가깝습니다. 에이전트는 한 번에 모든 걸 마법처럼 처리하는 존재라기보다, 입력 확인 → 산출물 생성 → 검증 → 반영 같은 경계를 분명히 해 줄 때 훨씬 안정적으로 움직입니다. 그리고 블로그처럼 매일 반복되는 작업일수록 이런 분리가 생각보다 큰 차이를 만듭니다.

저는 그래서 블로그 자동화에서 “한 번에 다 커밋하고 끝내기”보다, 콘텐츠 레포와 배포 레포를 분리해 두는 쪽을 강하게 선호합니다. 좀 더 boring해 보일 수는 있어도, 실제 운영에서는 boring한 구조가 훨씬 강합니다. OpenClaw는 이런 boring한 구조를 귀찮지 않게 실행해 주는 쪽에 강점이 있습니다. 매일 발행해야 하는 루틴이라면, 결국 중요한 건 화려함보다 어디서 실패했는지 바로 알 수 있는 구조입니다. 그 점에서 두 레포 구조는 꽤 실용적인 기본값입니다.
