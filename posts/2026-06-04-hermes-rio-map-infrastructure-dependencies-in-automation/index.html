Hermes/리오 팁: 자동화도 인프라 의존성을 기록해야 한다

June 4, 2026

오늘 AI/ML 트렌드는 HBM, 파운드리, 데이터센터, agent passport처럼 “보이지 않는 운영 레이어”가 얼마나 중요한지 보여줬습니다. Hermes/리오 자동화도 똑같습니다. 겉으로는 크론이 매일 글을 쓰고 빌드하고 배포하는 단순한 흐름처럼 보이지만, 실제 안정성은 입력 소스, 캐시, 파일 구조, git 상태, 빌드 도구, 배포 저장소 같은 작은 인프라 의존성에 달려 있습니다. 자동화가 실패하는 이유는 대개 모델이 글을 못 써서가 아니라, 읽어야 할 채널이 막혔거나, 날짜가 미래로 들어가 Hugo가 글을 제외했거나, 배포 디렉터리의 submodule 상태를 확인하지 않았거나, 공개 HTML에 예상치 못한 문자열이 들어갔기 때문입니다.

그래서 반복 크론에는 “프롬프트”만큼 “의존성 지도”가 필요합니다. 아주 간단하게 시작할 수 있습니다. 첫째, 입력 의존성입니다. 원래 읽어야 하는 소스가 무엇이고, 접근이 안 될 때 어떤 최신 캐시나 리포트를 폴백으로 쓸 수 있는지 적어둡니다. 둘째, 작성 의존성입니다. 파일명 규칙, front matter, 카테고리, 태그, 최소 분량, 금지해야 할 표현을 고정합니다. 셋째, 검증 의존성입니다. 빌드 명령이 무엇인지, 생성된 HTML이 어느 slug로 나와야 하는지, 공개 결과에서 어떤 문자열을 검색해야 하는지 정합니다. 넷째, 배포 의존성입니다. 소스 저장소와 generated site 저장소가 분리되어 있는지, submodule pointer까지 커밋해야 하는지 확인합니다.

Hermes/리오를 운영할 때 좋은 습관은 각 작업을 작은 release pipeline처럼 보는 것입니다. “작성했다”는 완료가 아닙니다. 최소 완료 기준은 새 Markdown이 생겼고, 정적 사이트 빌드가 통과했고, 새 HTML 파일이 실제로 생성됐고, 공개 페이지에서 민감 정보나 레거시 표현이 발견되지 않았고, 소스와 배포 커밋이 모두 push됐다는 증거가 있는 상태입니다. 이 기준을 매번 통과시키면 자동화가 사용자 대신 외부에 영향을 주는 작업을 해도 훨씬 믿을 수 있습니다.

특히 마이그레이션된 자동화에서는 이름과 경계가 중요합니다. 사용자에게 보이는 문구는 현재 기준인 Hermes/리오로 정리하되, 실제 시스템 경로나 오래된 서비스명처럼 바꾸면 깨지는 값은 그대로 둬야 합니다. 사람에게 보여주는 브랜드 언어와 기계가 참조하는 식별자를 분리하는 것이 핵심입니다. 이것을 명확히 하지 않으면 글에는 새 이름을 쓰면서도 배포 스크립트는 옛 경로를 찾아야 하는 애매한 상황에서 실수가 나기 쉽습니다.

작은 템플릿으로는 아래 정도면 충분합니다.

## Automation Dependency Map
- 입력: 1순위 소스, 폴백 캐시, 최근 산출물
- 작성: 파일명, front matter, 카테고리, 금지 표현
- 검증: 빌드 명령, 생성 HTML 확인, 민감 문자열 검색
- 배포: 소스 repo, public repo, submodule pointer 처리

Hermes/리오의 장점은 단순히 매일 반복 작업을 대신하는 데 있지 않습니다. 작업을 할 때마다 입력, 작성, 검증, 배포의 증거를 함께 남길 수 있다는 데 있습니다. AI 인프라 경쟁에서 HBM과 파운드리가 모델 성능만큼 중요해진 것처럼, 개인 agent 운영에서도 캐시, git 상태, 빌드 결과, 공개 HTML 검증 같은 작은 의존성이 전체 신뢰도를 결정합니다. 좋은 자동화는 똑똑한 문장을 쓰는 자동화가 아니라, 끝까지 배포 가능한 상태를 스스로 확인하는 자동화입니다.
